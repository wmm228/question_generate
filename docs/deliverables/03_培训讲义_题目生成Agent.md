# 题目生成 Agent 培训讲义

## 1. 培训目标

本讲义用于支持项目汇报、线上培训或 PPT 制作。培训对象可以是教师、教研人员、产品经理、平台运营人员和研发同学。

培训结束后，听众应能理解：

1. 题目生成 Agent 解决什么问题。
2. 它和普通大模型出题有什么区别。
3. 教师如何使用系统生成题目。
4. 系统如何保证题目质量。
5. 技术上如何接入 Tutor 和 OAH。
6. 交付材料中各文件的用途。

## 2. 建议培训时长

建议总时长：45 到 60 分钟。

| 模块 | 时长 | 内容 |
| --- | --- | --- |
| 项目背景 | 5 分钟 | 为什么需要题目生成 Agent |
| 功能演示 | 15 分钟 | 文本题、图片题、导出、历史 |
| Agent 流程 | 10 分钟 | 主 Agent、子 Agent、工具 |
| 案例讲解 | 10 分钟 | 典型案例 |
| 技术接入 | 10 分钟 | Tutor、OAH、部署 |
| 问答 | 5 分钟 | 常见问题 |

## 3. PPT 结构建议

### 第 1 页：标题页

标题：

```text
EDUQG 题目生成 Agent 项目介绍
```

副标题：

```text
面向教师出题、题库建设和教育平台接入的智能题目生成系统
```

讲解要点：

1. 这是一个可运行、可接入、可导出的题目生成服务。
2. 当前重点能力是题目生成 Agent，而不是单独展示某个模型。
3. 系统已经与 Tutor 服务和 OAH runtime 对齐。

### 第 2 页：为什么需要这个系统

页面内容：

1. 教师出题耗时。
2. 难度和知识点控制不稳定。
3. 图文题制作成本高。
4. 大模型直接生成容易格式不稳定。
5. 生成后缺少质量检查。

讲解词：

```text
普通大模型可以生成题目，但无法稳定保证学科、知识点、题型、难度、配图、答案和解析都符合教学要求。因此我们把出题过程做成一个 Agent 流程，先规范需求，再生成题目，再评价结果，最后输出结构化内容。
```

### 第 3 页：系统整体定位

页面内容：

```text
教师需求 -> 出题规范 -> Agent 调度 -> 题目生成 -> 质量评价 -> 导出/入库/沉淀
```

讲解要点：

1. 教师可以自然语言输入。
2. 系统将需求转成结构化规范。
3. OAH Agent 负责调度和生成。
4. Tutor 前端负责展示、历史、导出和反馈。

### 第 4 页：核心用户

页面内容：

| 用户 | 使用方式 |
| --- | --- |
| 教师 | 生成课堂练习、作业和小测 |
| 教研人员 | 批量构建题库 |
| 平台运营 | 接入智能作业和练习系统 |
| 学生系统 | 个性化练习和错题变式 |

讲解要点：

这个系统不是只给研发看的工具，最终目标是让教师和平台可以直接使用。

### 第 5 页：主要功能一览

页面内容：

1. 自然语言出题。
2. 文本题和图片题。
3. 选择题、判断题、简答题。
4. 多种算法策略。
5. 教师画像和学生画像。
6. 自动评价。
7. 历史记录。
8. Word/PDF/Excel 导出。

### 第 6 页：教师使用流程

页面内容：

```text
登录系统
  -> 新建出题会话
  -> 输入需求
  -> 确认关键字段
  -> 生成题目
  -> 查看评价
  -> 反馈或导出
```

讲解要点：

强调“确认关键字段”是系统稳定性的核心，不能让模型随意猜。

### 第 7 页：关键字段

页面内容：

| 字段 | 是否必须明确 |
| --- | --- |
| 学科 | 是 |
| 知识点 | 是 |
| 难度 | 是 |
| 题型 | 是 |
| 文本题/图片题 | 是 |
| 图片要求 | 图片题必须明确 |
| 算法 | 可默认 |

讲解词：

```text
这些字段属于教师控制字段。Agent 可以补充生成计划和解析表达，但不能随意改变教师选择的学科、知识点、难度和题型。
```

### 第 8 页：Agent 架构

页面内容：

```text
question-orchestrator
  -> spec-normalizer
  -> intent-recognizer
  -> text-question-generator
  -> visual-question-generator
  -> text-question-evaluator
  -> visual-question-evaluator
  -> student-simulator
  -> profile-evolution
```

讲解要点：

1. 主 Agent 负责理解需求和调度。
2. `spec-normalizer` 和 `intent-recognizer` 分别负责规范整理和生成授权判断。
3. 文本题和图片题走不同生成器。
4. 生成后必须经过评价器。
5. `student-simulator` 和 `profile-evolution` 用于学生画像和个性化沉淀。

### 第 9 页：工具链

页面内容：

| 工具 | 作用 |
| --- | --- |
| `validate_question_spec` | 校验题目规范 |
| `generate_visual_question` | 生成图片题草稿 |
| `run_evoq_text_question` | 高质量进化式文本出题 |
| `render_question_image` | 渲染题目图片 |
| `simulate_student_response` | 模拟学生作答 |
| `evaluate_text_question` | 评价文本题 |
| `evaluate_visual_question` | 评价图片题 |
| `read_profile` | 读取画像 |
| `write_profile` | 写入画像 |

讲解要点：

Agent 负责判断和组织流程，工具负责稳定的输入输出操作。

### 第 10 页：文本题案例

页面内容：

```text
需求：生成一道初中数学一次函数选择题，难度中等。
结果：题干、四个选项、答案、解析步骤。
```

讲解重点：

1. 是否围绕指定知识点。
2. 是否四个选项。
3. 是否唯一答案。
4. 解析是否完整。

### 第 11 页：图片题案例

页面内容：

```text
需求：生成一道勾股定理题，需要题干配图。
结果：题干 + 几何图 + 选项 + 答案 + 解析。
```

讲解重点：

图片题的核心不是“有图”，而是图片必须参与解题。

### 第 12 页：错题变式案例

页面内容：

```text
学生常错点：一次函数斜率和截距。
生成目标：同类变式，不重复原题数字。
```

讲解重点：

学生画像用于提示常见错误，帮助生成更有针对性的题。

### 第 13 页：质量评价

页面内容：

1. 知识点匹配。
2. 难度匹配。
3. 题型合规。
4. 答案正确。
5. 解析完整。
6. 图文一致。
7. 图片有效。

讲解要点：

生成不是结束，评价才决定题目能不能展示、保存或入库。

### 第 14 页：导出和交付

页面内容：

1. 题目结果导出：`docx`、`pdf`、`xls`。
2. 对话记录导出：用于追踪沟通过程。
3. 技术报告：用于研发和部署。
4. 案例集：用于演示和验收。

讲解提醒：

Word 图片缺失通常和旧式 HTML `.doc` 导出有关。正式题目结果应使用真 `.docx`。

### 第 15 页：技术部署概览

页面内容：

```text
Browser
  -> Tutor Web/Express
  -> OAH API
  -> question-orchestrator runtime
  -> model/tool chain
```

讲解要点：

1. Tutor 是 Web 服务和业务入口。
2. OAH 是 Agent runtime。
3. 模型配置通过环境变量和 OAH workspace 管理。
4. 远程 OAH 模式下，需要保证 Tutor 能访问 `OAH_BASE_URL`。
5. 完整同机部署时，仓库内置 `vendor/open-agent-harness`，可通过 `docker compose up -d --build` 同时启动 Tutor、OAH 和 PostgreSQL。

### 第 16 页：上线前检查

页面内容：

1. OAH 是否可访问。
2. workspace/runtime 是否存在。
3. `question-orchestrator` 是否可用。
4. 模型名称是否正确。
5. 文件存储目录是否可写。
6. 导出是否包含图片。

### 第 17 页：常见问题

问题一：为什么系统要追问？

回答：关键字段不完整时直接生成会导致题目偏离需求。

问题二：为什么图片题不能只放装饰图？

回答：教育题中的图片必须服务于作答或解析。

问题三：为什么导出的 Word 没图？

回答：如果是 HTML `.doc` 或 `data:image` 被转换工具清理，就会丢图。应使用真 `.docx`。

### 第 18 页：总结

页面内容：

```text
题目生成 Agent = 需求规范化 + Agent 调度 + 题目生成 + 自动评价 + 结果导出 + 历史沉淀
```

讲解词：

```text
这个项目的价值不在于单次生成一题，而在于把题目生产流程做成可控、可评估、可迁移的系统能力。
```

## 4. 现场演示脚本

### 演示一：文本题

输入：

```text
生成一道初中数学一次函数选择题，难度中等，不需要图片，解析要分步骤。
```

预期展示：

1. 系统识别题目规范。
2. 生成选择题。
3. 展示答案和解析。
4. 导出题目。

### 演示二：图片题

输入：

```text
生成一道初中数学勾股定理图片题，题干需要配图，选择题，难度 3。
```

预期展示：

1. 图片出现在题干区域。
2. 图片标注和题干一致。
3. 正确答案可由图片和题干推出。
4. 导出后图片可见。

### 演示三：错题变式

输入：

```text
学生对一次函数斜率判断容易错，生成一道变式题，不要重复原题。
```

预期展示：

1. 系统利用学生画像信号。
2. 生成同类但不重复的题目。
3. 干扰项体现常见错误。

## 5. 培训后作业

建议让参训人员完成三个操作：

1. 生成一道文本选择题并导出。
2. 生成一道图片题并检查图片是否参与作答。
3. 修改同一需求的难度，观察题目变化。

## 6. 培训验收标准

培训结束后，参训人员应能回答：

1. 什么是教师控制字段。
2. 为什么需要 Agent 调度。
3. 文本题和图片题的处理区别。
4. 题目质量评价检查哪些维度。
5. Word 图片丢失通常是什么原因。
