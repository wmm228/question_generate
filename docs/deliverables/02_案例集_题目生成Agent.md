# 题目生成 Agent 案例集

## 1. 案例集说明

本文档用于展示题目生成 Agent 的典型使用场景。案例覆盖文本题、图片题、错题变式、分层练习、EvoQ 高质量生成和平台导出等流程。

每个案例包含：

1. 使用场景。
2. 教师输入。
3. 系统识别的规范。
4. Agent 处理流程。
5. 示例输出。
6. 验收重点。

## 2. 案例一：初中数学勾股定理图片选择题

### 2.1 使用场景

教师希望在课堂小测中生成一道需要看图作答的勾股定理选择题。

### 2.2 教师输入

```text
帮我生成一道初中数学勾股定理单选题，难度中等，需要题干配图，解析要分步骤。
```

### 2.3 系统识别规范

| 字段 | 识别结果 |
| --- | --- |
| 学科 | 数学 |
| 学段 | 初中 |
| 知识点 | 勾股定理 |
| 题型 | 单选题 |
| 难度 | 3 / 中等 |
| 内容模式 | 图片题 |
| 图片位置 | 题干配图 |
| 图片要求 | 必须参与作答 |
| 出题算法 | direct 或 cot |

### 2.4 Agent 处理流程

1. `question-orchestrator` 判断需求字段完整。
2. `validate_question_spec` 校验题目规范。
3. `visual-question-generator` 生成题干、选项、答案、解析和配图说明。
4. `render_question_image` 生成题干配图。
5. `visual-question-evaluator` 检查图文一致性、答案正确性和图片可用性。
6. 系统返回结构化结果。

### 2.5 示例输出

题干：

```text
如图，在直角三角形 ABC 中，∠C = 90°，AC = 6 cm，BC = 8 cm。求斜边 AB 的长度。
```

选项：

```text
A. 8 cm
B. 10 cm
C. 12 cm
D. 14 cm
```

正确答案：

```text
B
```

解析：

```text
1. 因为 ∠C = 90°，所以 AB 是斜边。
2. 根据勾股定理，AB² = AC² + BC²。
3. 代入 AC = 6，BC = 8，得到 AB² = 36 + 64 = 100。
4. 所以 AB = 10 cm。
```

图片要求：

```text
题干图中必须显示直角三角形 ABC，并标注 AC = 6 cm、BC = 8 cm、∠C = 90°。
```

### 2.6 验收重点

1. 图片是否包含解题必要信息。
2. 题干数据和图片标注是否一致。
3. 选项是否有唯一正确答案。
4. 解析是否能推出答案。

## 3. 案例二：高中物理受力分析题

### 3.1 使用场景

教师希望生成一道受力分析简答题，用于课堂讲解。

### 3.2 教师输入

```text
生成一道高中物理牛顿第二定律相关的受力分析题，难度偏基础，需要配图，要求学生写出加速度大小。
```

### 3.3 系统识别规范

| 字段 | 识别结果 |
| --- | --- |
| 学科 | 物理 |
| 学段 | 高中 |
| 知识点 | 牛顿第二定律、受力分析 |
| 题型 | 简答题 |
| 难度 | 2 |
| 内容模式 | 图片题 |
| 图片位置 | 题干配图 |
| 出题算法 | cot |

### 3.4 Agent 处理流程

1. 主 Agent 确认学科、题型、难度和图片要求。
2. 图片生成子 Agent 构造物块、水平面、拉力、摩擦力方向。
3. 评价子 Agent 检查受力方向和数值是否一致。
4. 输出题目和分步解析。

### 3.5 示例输出

题干：

```text
如图，一个质量为 2 kg 的物块放在光滑水平面上，受到水平向右的恒力 F = 10 N。求物块的加速度大小。
```

标准答案：

```text
5 m/s²
```

解析：

```text
1. 水平面光滑，忽略摩擦力。
2. 水平方向合力等于 F = 10 N。
3. 根据牛顿第二定律 F = ma。
4. a = F / m = 10 / 2 = 5 m/s²。
```

### 3.6 验收重点

1. 图片是否清楚显示物块和力的方向。
2. 是否避免把“光滑水平面”写成有摩擦。
3. 单位是否正确。
4. 解析是否符合牛顿第二定律。

## 4. 案例三：错题变式生成

### 4.1 使用场景

学生在一次函数图像题中出错，教师希望生成同类型但不重复的新题。

### 4.2 教师输入

```text
学生在一次函数图像和斜率判断上容易错。帮我生成一道同类变式题，不要和原题数字一样，难度中等，选择题。
```

### 4.3 系统识别规范

| 字段 | 识别结果 |
| --- | --- |
| 学科 | 数学 |
| 知识点 | 一次函数图像、斜率 |
| 题型 | 单选题 |
| 难度 | 3 |
| 内容模式 | 文本题或图片题 |
| 学生画像信号 | 斜率判断易错 |
| 出题算法 | eqpr 或 evoq |

### 4.4 Agent 处理流程

1. 主 Agent 提取学生错因。
2. 生成 Agent 设计同类变式。
3. 评价 Agent 检查新题是否与原题过度相似。
4. 若使用 EvoQ，则生成多个候选并择优。

### 4.5 示例输出

题干：

```text
已知一次函数 y = -2x + 5。下列说法正确的是哪一项？
```

选项：

```text
A. 函数图像经过第一、二、三象限
B. 函数图像随 x 增大而增大
C. 函数图像与 y 轴交于点 (0, 5)
D. 函数图像的斜率为 5
```

正确答案：

```text
C
```

解析：

```text
1. 一次函数 y = kx + b 中，k 表示斜率，b 表示 y 轴截距。
2. 本题 k = -2，b = 5。
3. 因此图像与 y 轴交于 (0, 5)。
4. k < 0，所以函数随 x 增大而减小。
```

### 4.6 验收重点

1. 是否针对“斜率”和“截距”两个易错点。
2. 是否与原题不同但考查同类能力。
3. 干扰项是否有诊断意义。

## 5. 案例四：分层练习生成

### 5.1 使用场景

教师希望同一知识点生成基础、中等、提升三档题。

### 5.2 教师输入

```text
围绕初中数学二元一次方程组，分别生成基础、中等、提升三道选择题，用于分层练习。
```

### 5.3 系统识别规范

| 字段 | 识别结果 |
| --- | --- |
| 学科 | 数学 |
| 知识点 | 二元一次方程组 |
| 题型 | 单选题 |
| 难度 | 2、3、4 |
| 内容模式 | 文本题 |
| 出题算法 | direct / cot |

### 5.4 示例输出方向

基础题：

```text
直接代入或消元，计算一组整数解。
```

中等题：

```text
根据实际情境列方程组，再求解。
```

提升题：

```text
包含参数或条件判断，需要先分析再计算。
```

### 5.5 验收重点

1. 三道题是否围绕同一知识点。
2. 难度梯度是否明显。
3. 是否避免只是改数字。
4. 是否适合不同水平学生。

## 6. 案例五：平台题库批量入库

### 6.1 使用场景

教育平台需要围绕某章节生成一批可入库的题目。

### 6.2 平台输入

```json
{
  "subject": "数学",
  "knowledge_point": "三角函数基础",
  "difficulty": "3",
  "question_type": "multiple_choice",
  "content_mode": "text",
  "algorithm": "eqpr"
}
```

### 6.3 系统输出结构

```json
{
  "question": "题干文本",
  "options": ["A. ...", "B. ...", "C. ...", "D. ..."],
  "ground_truth": "B",
  "solution_steps": ["步骤一", "步骤二"],
  "meta": {
    "subject": "数学",
    "question_type": "multiple_choice",
    "content_mode": "text"
  },
  "visual_pipeline": {
    "requested": false,
    "status": "not_requested"
  }
}
```

### 6.4 验收重点

1. JSON 字段是否稳定。
2. 选项数量是否固定。
3. 答案格式是否可机器读取。
4. 是否能保存到题库。

## 7. 案例六：图片导出问题复现

### 7.1 场景

用户下载 Word 后发现题目说“下图”，但文档里看不到图片。

### 7.2 现象

某些下载文件看起来是 `.doc`，但实际内容是 HTML：

```html
<!doctype html>
<html>
  <body>
    <p>题干</p>
    <p>下图为...</p>
    <p><img src="data:image/png;base64,..."></p>
  </body>
</html>
```

### 7.3 原因

这不是标准 `.docx`。标准 `.docx` 应该是 zip 包结构，图片存放在 `word/media/` 下，并通过 `word/_rels/document.xml.rels` 建立关系。HTML 伪装的 `.doc` 使用 `data:image` 嵌入图片，部分 Word、浏览器、微信和在线预览工具会忽略或清理这种图片。

### 7.4 正确处理

1. 题目结果导出应使用真正 `.docx`。
2. 图片应作为二进制文件写入 `word/media/`。
3. `document.xml` 中使用 `r:embed` 引用图片关系。
4. 前端下载文件扩展名和后端 Content-Type 必须一致。
