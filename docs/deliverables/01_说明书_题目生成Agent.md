# 题目生成 Agent 说明书

## 1. 文档定位

本文档用于说明当前题目生成 Agent 的产品定位、适用对象、核心功能、使用流程、输入输出、导出方式和运行注意事项。

当前系统不是一个单纯的“大模型出题页面”，而是一个面向教师、教研人员和教育平台的智能出题服务。它把教师的自然语言需求、页面表单参数、教师画像、学生画像和 OAH Agent 调用流程组织起来，输出可展示、可审核、可导出、可沉淀的结构化题目。

## 2. 系统名称与交付对象

系统名称：EDUQG 题目生成 Agent

主要交付对象：

1. 教师：用于课堂练习、课后作业、单元检测、错题变式和分层练习。
2. 教研人员：用于批量构建题库、设计测评内容、验证知识点覆盖。
3. 教育平台：用于接入题库系统、智能作业系统、AI 伴学系统和个性化练习系统。
4. 研发和部署人员：用于把题目生成能力接入 Tutor、OAH 或后续平台。

## 3. 系统目标

题目生成 Agent 的目标可以概括为四点。

1. 让出题需求结构化。

教师可以用自然语言描述需求，也可以通过页面字段选择参数。系统会把需求整理成统一的题目生成规范，避免“只凭一句话让模型自由发挥”。

2. 让生成过程可控。

学科、知识点、难度、题型、是否配图、出题算法等关键字段必须来自教师或业务默认值。Agent 可以建议缺失字段，但不能静默修改教师明确选择的核心条件。

3. 让题目结果可用。

最终输出不只是题干，还包括选项、正确答案、解析步骤、图片资源、结构化内容、质量评价、生成元数据和历史记录。

4. 让系统可以迁移部署。

当前实现已经接入 Tutor 服务和 OAH runtime，可以通过 Web 页面、HTTP API、OAH workspace 和部署配置进行调用。

## 4. 核心功能

### 4.1 教师需求理解

教师可以输入类似下面的需求：

```text
帮我出一道初中数学勾股定理单选题，难度中等，需要配图，解析要分步骤。
```

系统需要识别：

| 字段 | 示例 |
| --- | --- |
| 学科 | 数学 |
| 知识点 | 勾股定理 |
| 难度 | 中等 / 3 |
| 题型 | 单选题 |
| 内容模式 | 图片题 |
| 配图位置 | 题干配图 |
| 解析要求 | 分步骤 |

### 4.2 出题规范确认

系统会把需求整理成 `edu-question-spec.v1` 风格的题目规范。关键字段包括：

| 字段 | 含义 |
| --- | --- |
| `subject` | 学科 |
| `knowledge_point` | 知识点 |
| `difficulty` | 难度 |
| `question_type` | 题型 |
| `content_mode` | 文本题或图片题 |
| `algorithm` | 出题算法 |
| `image_requirement` | 图片模式、图片位置和图片是否参与作答 |

如果关键字段缺失，系统会先追问或阻止生成，而不是直接调用模型。

### 4.3 多种出题策略

当前支持的算法包括：

| 算法 | 中文说明 | 适用场景 |
| --- | --- | --- |
| `direct` | 直接生成 | 快速生成普通题 |
| `cot` | 分步推理 | 需要强调推理过程的题 |
| `react` | 推理-行动-观察 | 需要多步检查和修正的题 |
| `dear` | 分解分析修订 | 需要拆解任务、逐步改写的题 |
| `eqpr` | 评估起草处理修订 | 需要生成后评价和优化的题 |
| `evoq` | 进化式出题 | 需要多轮候选、选择和改进的高质量题 |

### 4.4 文本题生成

文本题适用于不需要图片参与作答的题目。系统输出题干、选项、答案和解析，适合选择题、判断题和简答题。

### 4.5 图片题生成

图片题适用于图形、图像、受力图、电路图、平面图、坐标图等必须依赖视觉信息的题目。

图片题要求：

1. 图片必须与题目条件一致。
2. 图片必须参与作答或解析。
3. 图片不能只是装饰。
4. 图片题需要经过图文一致性和可渲染性检查。

当前系统支持题干配图、选项配图和解析配图三类位置。

### 4.6 教师画像和学生画像

教师画像用于保存教师偏好，例如：

1. 常用学科和学段。
2. 喜欢的题目风格。
3. 难度控制方式。
4. 是否偏好分步骤解析。
5. 是否偏好图文题。

学生画像用于保存学生学习状态，例如：

1. 能力水平。
2. 知识点掌握度。
3. 常见错误。
4. 学习偏好。
5. 历史表现。

画像是上下文信号，不会覆盖教师本轮明确输入的出题要求。

### 4.7 自动评价

系统会对生成结果进行基础质量检查，重点包括：

| 维度 | 检查内容 |
| --- | --- |
| 知识点匹配 | 是否围绕指定知识点 |
| 难度匹配 | 是否符合教师设定难度 |
| 题型合规 | 选择题是否有四个选项和唯一答案 |
| 答案正确 | 解析是否能推出标准答案 |
| 解析完整 | 解题步骤是否清楚 |
| 图文一致 | 图片内容是否和题干一致 |
| 图片有效 | 图片是否参与作答 |

### 4.8 历史记录和题库沉淀

系统保存出题会话、生成结果、用户反馈和题目历史。教师可以查看历史生成题，平台也可以基于这些结果做题库沉淀和质量追踪。

### 4.9 导出能力

当前题目结果支持导出为：

1. `docx`：适合交给教师二次编辑或归档。
2. `pdf`：适合预览、打印、分发。
3. `xls`：适合简单表格化保存。

注意：题目结果导出和对话记录导出不是同一个出口。题目结果导出面向题目内容；对话记录导出面向沟通过程。

## 5. 标准使用流程

### 5.1 进入系统

用户登录 Tutor 系统后进入题目生成工作台。

### 5.2 创建或选择画像

教师可以新建试题描述规范，也可以继续历史会话。系统会保留近期对话和压缩后的会话记忆。

### 5.3 输入出题需求

教师可以直接输入自然语言，也可以通过页面字段设置：

1. 学科。
2. 知识点。
3. 难度。
4. 题型。
5. 文本题或图片题。
6. 图片位置。
7. 出题算法。

### 5.4 系统确认规范

系统检查规范是否完整。如果缺少必填项，会提示教师补充。例如：

```text
请确认题型和难度后再生成。
```

### 5.5 调用 Agent 生成题目

规范就绪后，系统调用 OAH 中的 `question-orchestrator`，由主 Agent 选择对应子 Agent 和工具链。

### 5.6 查看结果

教师可以查看：

1. 题干。
2. 选项。
3. 正确答案。
4. 解题步骤。
5. 配图。
6. 质量评价。
7. 生成元数据。

### 5.7 反馈和导出

教师可以对题目打分或重新生成，也可以导出题目结果。

## 6. Agent 架构说明

当前系统采用主 Agent + 子 Agent 的结构。

| 角色 | 职责 |
| --- | --- |
| `question-orchestrator` | 主调度 Agent，负责教师对话、规范确认和路由 |
| `spec-normalizer` | 将教师自然语言或表单输入整理为 `edu-question-spec.v1` |
| `intent-recognizer` | 独立判断教师当前轮是否授权立即生成题目 |
| `text-question-generator` | 生成文本题 |
| `visual-question-generator` | 生成图片题和视觉依赖 |
| `text-question-evaluator` | 评价文本题质量 |
| `visual-question-evaluator` | 评价图片题、图片相关性和渲染安全 |
| `student-simulator` | 根据学生画像模拟可能作答和错误倾向 |
| `profile-evolution` | 生成教师画像和学生画像的更新建议 |

主 Agent 负责调度，不直接替教师改核心字段。8 个子 Agent 分别承担规范化、意图判断、生成、评价、学生模拟和画像演化职责，不能在实现中合并成单一生成器。

## 7. 工具与规则

当前 OAH runtime 合约中的核心工具包括：

| 工具 | 用途 |
| --- | --- |
| `validate_question_spec` | 校验题目生成规范 |
| `generate_visual_question` | 生成图片题草稿和视觉依赖 |
| `run_evoq_text_question` | 执行 EvoQ 文本题进化式生成 |
| `render_question_image` | 渲染或返回题目图片资源 |
| `simulate_student_response` | 模拟学生作答和常见错误 |
| `evaluate_text_question` | 评价文本题质量 |
| `evaluate_visual_question` | 评价图片题质量、图文一致和渲染安全 |
| `read_profile` | 读取教师或学生画像信号 |
| `write_profile` | 写入画像更新建议或状态 |

核心规则：

1. 文本题不能包含图片目标。
2. 图片题必须包含至少一个图片目标。
3. 图片题的视觉信息必须参与作答。
4. 单选题必须有四个选项和唯一正确答案。
5. AI 输出必须经过解析和结构校验后再展示。

## 8. 部署与运行说明

当前系统主要由两部分组成：

1. Tutor：Node.js + Express Web 服务。
2. OAH：Agent runtime 和模型调用服务。

Tutor 侧关键环境变量：

```env
OAH_BASE_URL=http://YOUR_OAH_HOST:8787
OAH_AGENT_NAME=question-orchestrator
OAH_INTENT_AGENT_NAME=intent-recognizer
OAH_WORKSPACE_RUNTIME=tutor-question-generation
OAH_WORKSPACE_NAME=tutor-question-generation
OAH_WORKSPACE_OWNER_ID=tutor
OAH_WORKSPACE_AUTO_CREATE=true
OAH_MODEL_NAME=platform/your-model
TUTOR_PORT=7896
```

常用命令：

```bash
npm install
npm run build
npm run start
```

开发调试：

```bash
npm run dev
```

## 9. 常见问题

### 9.1 为什么缺少字段时不能直接生成

因为学科、知识点、难度、题型、内容模式和图片要求属于教师控制字段。系统不能替教师静默决定，否则会导致题目偏离教学目标。

### 9.2 为什么图片题必须检查图片相关性

教育题目中的图片不是装饰物。如果题干说“根据下图”，图片必须提供解题所需信息，否则学生无法作答，题目不可用。

### 9.3 为什么同样的需求可能生成不同题目

大模型生成具有一定随机性；不同算法、模型、上下文和历史画像都会影响结果。系统通过规范校验、评价和反馈机制降低不稳定性。

### 9.4 为什么 Word 下载有时没有图片

如果下载文件是 HTML 伪装成 `.doc`，并且图片写成 `data:image/png;base64,...`，部分 Word、浏览器、微信或转换工具会忽略这种图片。稳定做法是导出真正的 `.docx`，把图片写入 `word/media/` 并在 `word/_rels/document.xml.rels` 中建立图片关系。

当前题目结果导出已经有 `.docx` 路径；如果拿到的是 `.doc` 文件，通常说明走的是旧的 HTML Word 导出或对话记录导出路径。

## 10. 验收要点

验收时建议检查：

1. 能否登录并进入题目生成工作台。
2. 能否创建出题会话。
3. 缺字段时是否会追问或阻止生成。
4. 文本题能否生成题干、选项、答案、解析。
5. 图片题能否生成可见且相关的图片。
6. 生成结果是否能导出。
7. OAH 状态诊断是否能显示模型和 workspace 状态。
8. 历史记录、画像和反馈是否可保存。
